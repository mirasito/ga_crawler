viled.kz — Anti-Bot защита: что построить и в каком порядке

Документ написан 2026-05-06 на основе практики Phase 1 spike (parsing viled.kz vs goldapple.kz) и Phase 3 production-кода ga_crawler. Точка зрения: тот, кто только что обошёл goldapple, смотрит на viled и видит как его обходить.

TL;DR — что увидит атакующий сегодня viled.kz отдаёт весь товарный каталог (бренды, цены, остатки, описания) одним обычным GET-запросом через curl: данные лежат в __NEXT_DATA__ JSON-блоке прямо в HTML. Никакого TLS-fingerprint'а, никакого rate-limit, никакой geo-проверки, никаких капчей. 15/15 успешных запросов через curl_cffi в Phase 1 spike, скорость скрейпа ограничена только вежливостью атакующего, не вашей защитой. На спайке мы поняли: чтобы дойти до уровня goldapple — у нас впереди примерно 4 года инженерных решений, которые они уже приняли. У вас этого нет совсем.

Контекст: что мы знаем из ga_crawler

В рамках проекта ga_crawler команда viled запустила еженедельный краулер прайс-конкуренции — парсит и goldapple, и viled, потом сводит их в Excel. Phase 1 spike (январь–май 2026) измерил обе стороны эмпирически. Цифры ниже — не теория, а наш собственный production opyt:

СайтСтек скрейпераУспехЧто нужноСтоимость скрейпинга
viled.kz curl_cffi + selectolax + __NEXT_DATA__ 15/15 (100%) Любая машина, любой IP ~$0/нед, ~10 мин кода
goldapple.kz Camoufox 135.0.1-beta.24 (Firefox + C++ fingerprint spoof) + KZ-IP + 3-5 сек random pause + per-SKU fresh profile + retry+jitter + smoke probe 99/100 (но требует cooldown'ы) Firefox-based anti-detect browser (~250 MB), KZ-residential IP, ~5.5 часа sequential на еженедельный run ~$0–2/нед прокси + ~1 неделя инженерного времени на спайк + Phase 3 (8 waves, 192 теста, ~50 коммитов)

Разница в стоимости обхода защиты — на два порядка. Это и есть разница «есть anti-bot vs нет».

1. Что атакующий видит на viled.kz прямо сейчас

Из Phase 1 spike findings (`.planning/spikes/01-goldapple/MEMO.md`) и Phase 2 контракта парсера:

1.1 Полный товарный JSON в HTML

curl -s https://viled.kz/p/some-product | grep -o '__NEXT_DATA__[^<]*' | head -1
# → весь товарный объект: бренд, категория, цена, sku, остатки, описание, фото-URLs

Next.js серверной отрисовкой вставляет в HTML блок <script id="__NEXT_DATA__">...</script>, содержащий весь state страницы как сериализованный JSON. Скрейперу не нужно парсить DOM, не нужно выполнять JS, не нужно даже знать селекторы — он просто json.loads() и получает структурированную базу.

Это самая большая проблема В __NEXT_DATA__ утекают поля, которых на самой странице нет визуально: внутренние SKU-id'ы, остатки на складах, метаданные категоризации, иногда — служебные цены/маржа. Атакующий получает больше данных, чем «обычный пользователь» видит в UI.

1.2 Никакой TLS/JA3 fingerprint-проверки

goldapple.kz в Phase 1 спайке возвращал HTTP 403 на запросы Python requests (urllib3 TLS fingerprint детектируется WAF'ом за миллисекунды). Решение для атакующего — curl_cffi с impersonate="chrome", который имитирует Chrome 136 TLS+JA3+HTTP/2 fingerprint один-в-один.

viled.kz отдаёт HTTP 200 на любой fingerprint, включая голый Python urllib3. То есть нет даже первого, самого дешёвого слоя проверки.

1.3 Никаких rate-limit'ов в наблюдаемом окне

В Phase 1 спайке мы делали 15 последовательных запросов к продуктовым страницам без задержек — все 15 вернули 200 OK. По косвенным данным (logs, response time consistent ~200ms), никакого throttling по IP / user-agent / session нет.

1.4 Sitemap и server-side data — публичны и полны

Если у viled есть /sitemap.xml (что у Next.js обычно по умолчанию), скрейпер за один запрос получает список всех URL'ов товаров. Никакой «discover-by-crawling» не нужен.

1.5 Что НЕ нашли

  • Cloudflare или другая CDN-защита уровня L7 — отсутствует
  • WAF (DataDome, F.A.C.C.T., Imperva, Akamai Bot Manager) — отсутствует
  • JavaScript challenge или CAPTCHA на товарных страницах — отсутствует
  • Behavioral analysis (mouse/touch/scroll patterns) — отсутствует
  • Geo-IP проверка — отсутствует
  • Authentication wall — отсутствует (что правильно: товары публичные)
  • Rate-limit на anonymous-traffic — отсутствует или слишком высокий

3. Тиерированные рекомендации

Не стоит сразу строить Tier 4. Большинство ROI в первых трёх тирах. Пройдитесь по ним сверху вниз; останавливайтесь, когда стоимость защиты превысит value сохранённых данных.

T0 Гигиена данных: убрать утечки в __NEXT_DATA__; robots.txt и noindex на служебные страницы; не коммитить служебные API-ключи в frontend bundle Самый дешёвый класс атакующих (студенты, junior-script-kiddies) 1–2 дня ~0₸ Нет — это просто гигиена

T2+ оправдан только если виледа реально начнут парсить конкуренты (которые делают то же что мы делаем для goldapple) — обнаружить это можно по аномалиям трафика, и тогда уже эскалировать.

4. Конкретный план действий (приоритезированный)

Неделя 1 — T0 гигиена данных

  • Аудит __NEXT_DATA__: в getServerSideProps/getStaticProps явно фильтруйте поля product-объекта перед сериализацией. Возвращайте только то, что реально нужно для рендера UI. Внутренние SKU-id'ы, складские остатки, маржа — оставить server-only через API с авторизацией.
  • Перейти с SSR на ISR + client-side data fetch для критичных полей. Цена и остатки получаются client-side через JSON API с rate-limit и Cloudflare challenge. Для SEO достаточно отдавать бренд + название + категорию серверно (это всё равно публичная информация в Google).
  • Robots.txt: 
    User-agent: *
Allow: /
Disallow: /api/
Disallow: /search?
Crawl-delay: 2

User-agent: GPTBot
Disallow: /

User-agent: ClaudeBot
Disallow: /

User-agent: PerplexityBot
Disallow: /

User-agent: SemrushBot
Disallow: /

User-agent: AhrefsBot
Disallow: /
    LLM/SEO-боты не должны индексировать ваш каталог в свои базы. Это не блокирует их технически (robots.txt — джентльменское соглашение), но даёт правовую базу для DMCA / cease-and-desist.
  • Скрытые служебные эндпоинты: /admin, /_next/data/*.json, /api/internal/* — ставьте либо за Basic Auth, либо отдавайте 404 на запросы без правильного Origin/Referer.

Неделя 2 — T1 базовая защита

  • Перенаправить DNS через Cloudflare (Free план).
  • Включить Bot Fight Mode в Cloudflare → Security → Bots.
  • Включить Managed Rules → OWASP Core Rule Set.
  • Настроить rate-limit: 
    (http.request.uri.path matches "^/p/.*")
  AND (cf.threat_score gt 14)
  → Managed Challenge

(http.request.uri.path matches "^/p/.*")
  AND (rate(60s) gt 30)
  → Block (1h)
  • Whitelist для поисковиков: убедитесь, что Cloudflare правильно идентифицирует Googlebot, YandexBot, Bingbot, DuckDuckBot — иначе SEO пострадает.
  • Логи в SIEM или просто в Sentry/Datadog: собирайте rate-limit hits и Bot Fight challenges. Будете видеть, откуда идут попытки.

Неделя 3 — Мониторинг

  • Поднимите дашборд: количество challenge'ей в день, уникальных IP с >100 product-page hits в час, distribution geo (KZ vs не-KZ).
  • Если видите устойчивые >1000 hits/день из одного ASN — это или crawler-conkurент, или прайс-агрегатор. В первом случае усиливайте до T2; во втором — добавляйте robots.txt и блокируйте по ASN.

Опционально, неделя 4+ — T2 если данные ползут

Триггер: видите устойчивый «голодный» трафик от датацентр-ASN'ов (DigitalOcean, OVH, Hetzner, AWS) на товарные страницы. Это значит, что кто-то парсит вас headless-Chromium'ом из VPS.

  • Cloudflare Pro/Business plan
  • JS Challenge на product-pages для не-search-bot трафика
  • Опционально — FingerprintJS Pro для distinguishing real users vs Patchright

5. Чего НЕ делать (антипаттерны)

Не блокируйте всё, что не похоже на Chrome 120 Реальные пользователи на Safari iOS, Firefox, Yandex Browser, старых Android Chrome — будут получать капчу. CTR упадёт. Подбирайте challenge thresholds по реальному трафику, а не по параноидальным дефолтам.
Не закрывайте sitemap.xml целиком Это убьёт SEO. Если беспокоитесь, что атакующий за один запрос получает все URL'ы — отдавайте sitemap, но дробите его на <50K URL'ов per-file (что хорошая практика для SEO в любом случае), и логируйте, кто его скачивает. Реально атакующие сами спарсят URL'ы из категорий, если их нет в sitemap; вы только усложните жизнь Google.
Не покупайте «AI-powered Bot Detection» по одному обещанию вендора Многие enterprise-вендоры обещают 99% детект, но в реальности их защита обходится Camoufox или ScrapingBee Web Unlocker за $1–2/1000 страниц. Перед покупкой просите PoC: возьмите наш ga_crawler stack (Camoufox + KZ-IP + правильные паузы), натравите на их demo, измерьте recall. Если >50% запросов проходит — продукт не защищает от того, что вы боитесь.
Не делайте custom anti-bot in-house без штатного security-инженера F.A.C.C.T. / DataDome держат команды по 50+ человек, обновляющих эвристики каждую неделю в ответ на новые anti-detect браузеры. In-house JS-fingerprint detector протянет 6 месяцев, потом устареет и даст ложное чувство защиты.
Не блокируйте по User-Agent — это бесполезно impersonate="chrome" в curl_cffi отдаёт идеальный Chrome UA. UA-блокировка ловит только тех, кто не умеет делать headers={'User-Agent': 'Mozilla/5.0...'} — то есть никого старше 18 лет. Это театр безопасности.

6. Как тестировать защиту

После каждого тира защиты — натравите наш собственный краулер на viled. Это идеальный red-team:

Тест 0 — наивный curl

curl -s "https://viled.kz/p/some-product" | grep '__NEXT_DATA__'
# Сейчас: возвращает JSON с данными
# После T0: должно вернуть пустой/обрезанный JSON
# После T1: возможно Cloudflare challenge (зависит от threat_score)

Тест 1 — curl_cffi с Chrome impersonation

uv run python -c "
from curl_cffi import requests
r = requests.get('https://viled.kz/p/some-product', impersonate='chrome')
print(r.status_code, len(r.text))
"
# Сейчас: 200, ~150KB
# После T1 Bot Fight: должно быть challenge HTML или 403
# После T2: точно challenge

Тест 2 — Playwright

uv run python -c "
import asyncio
from playwright.async_api import async_playwright

async def main():
    async with async_playwright() as p:
        browser = await p.chromium.launch(headless=True)
        page = await browser.new_page()
        r = await page.goto('https://viled.kz/p/some-product')
        print(r.status, len(await page.content()))
        await browser.close()

asyncio.run(main())
"
# После T2: Cloudflare JS challenge может пройти, может нет (зависит от threat score)
# После T2 + FingerprintJS: должно проваливаться

Тест 3 — Camoufox (наш «настоящий attacker»)

uv run python -c "
import asyncio
from camoufox.async_api import AsyncCamoufox

async def main():
    async with AsyncCamoufox(headless=True, geoip=True, humanize=True) as browser:
        page = await browser.new_page()
        r = await page.goto('https://viled.kz/p/some-product')
        print(r.status, len(await page.content()))

asyncio.run(main())
"
# Это серьёзный test — в Phase 1 spike Camoufox прошёл goldapple 99/100.
# Если viled после T1+T2 всё ещё пропускает Camoufox с KZ-IP — нужен T3.
# Если фильтрует — T1+T2 достаточны для viled на 2026.

Пройдите эти тесты после каждой настройки и фиксируйте baseline. Если завтра всё тот же тест вдруг проходит — атакующая сторона выпустила новую версию (Camoufox каждые 2-3 недели обновляется), и вам надо реагировать.

9. Что делать с тем, что мы знаем про goldapple

Поскольку наша команда теперь умеет обходить goldapple, есть legal/PR risk: если goldapple узнают, что viled.kz запускает еженедельный краулер их каталога, они могут эскалировать до cease-and-desist. Несколько защитных мер:

  • Документировать законность: Phase 1 spike зафиксировал, что мы парсим только публичные данные без логина, не нарушаем robots.txt и не превышаем разумный rate-limit. У нас есть аудит-trail в .planning/spikes/01-goldapple/MEMO.md и Phase 3 коде. Это defensible posture.
  • Запускать crawler с известного KZ-IP с человеческим UA: не маскировать как DDoS, не использовать residential proxy если можно обойтись без них (мы и не используем — Phase 1 verdict: Camoufox solves the gate без proxy).
  • Не публиковать данные goldapple: ga_crawler пишет всё в SQLite, который живёт на нашем VPS. Telegram-доставка идёт в приватный канал команды viled. Никаких public mirrors / API.
  • Готовить vsa-protocol: если goldapple пришлют письмо — ответить официальным изложением scope'а (price comparison, weekly cadence, no PII), предложить partnership вместо confront (часто работает в B2B retail).

10. Резюме

  1. Сейчас viled.kz атакуется одной строкой curl. Это нормально для startup, но ненормально для бизнеса с серьёзным конкурентом, который сам активно занимается prices monitoring.
  2. T0 + T1 за 2-3 недели поднимают барьер с «1 строка» до «нужно настраивать Playwright, обходить Cloudflare challenge, ловить rate-limit'ы». Это отсеет ~95% атакующих.
  3. T2+ оправдан только когда мы увидим в логах устойчивые попытки обойти T1. Преждевременная инвестиция в DataDome/F.A.C.C.T. — это $50K/год на ничто.
  4. Тестируйте свою защиту своим же ga_crawler stack'ом. Если ваша Camoufox-конфигурация для goldapple проходит viled — значит viled до уровня goldapple ещё далеко.
  5. Не забывайте: лучшая защита данных — не отдавать их клиенту. T0 (фильтрация __NEXT_DATA__) самая дешёвая и самая эффективная мера. Cloudflare можно обойти, fingerprint можно подделать, но если данных в HTML просто нет — их не достать никаким стеком.
Финальная рекомендация Запланируйте T0 на следующий sprint (1 backend + 0.5 PM, ~1 неделя). Запланируйте T1 на спринт после T0 (1 devops, ~3 дня). Поднимите Grafana-дашборд для challenge-метрик. Через 8 недель смотрите логи — если automated traffic вырос >500 unique IP/нед на product-pages, эскалируйте до T2. Иначе остаётесь на T0+T1 и тратите бюджет на функциональность, не на security театр.
