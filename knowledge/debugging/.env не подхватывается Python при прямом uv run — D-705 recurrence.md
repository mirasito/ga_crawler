# .env не подхватывается Python при прямом uv run — D-705 recurrence

## Симптом

Run #13 2026-05-13 12:09 UTC завершился `status=success` НО `delivery_status=skipped_no_credentials, last_error=missing_env_TG_BOT_TOKEN` — Telegram-доставка пропущена.

`.env` файл существует, заполнен валидными значениями (`TG_BOT_TOKEN`, `TG_BUSINESS_CHAT_ID=986299192`, `TG_OPS_CHAT_ID=986299192`, `HC_PING_URL`), `chmod 0600`.

## Root cause

Pipeline read env vars через `os.environ` напрямую, БЕЗ `python-dotenv.load_dotenv()` вызова. На production VPS это работает потому что `bin/weekly-run.sh` шаг 7 делает:

```bash
set -a
source .env
set +a
```

— экспортирует все KEY=VALUE как shell-environment перед `python -m ga_crawler weekly-run`.

Но при прямом запуске `uv run python -m ga_crawler weekly-run` (мой случай — Windows local, no bash wrapper) — shell-environment не имеет `TG_*` vars. Python `os.environ.get("TG_BOT_TOKEN")` → None → delivery gate trips → `skipped_no_credentials`.

Это **D-705 design decision из Phase 7**: "bash wrapper владеет env loading, Python child работает только с уже-loaded env" — оптимизация для cron path. Но создаёт failure mode для:
1. Local dev на Windows (bash wrapper не запустить)
2. Manual recovery через `python -m ga_crawler deliver-run --run-id N`
3. Любой code path который bypass'ит bash wrapper

## Recurrence (это уже 2-я встреча)

Phase 7 SECURITY audit document T-07-08 mitigation `.env chmod 0600` уже упомянул D-705 — но не зафиксировал это как **architectural bug**. Только как documented invariant. Run #13 показал что invariant нарушается при первом же deviation от cron path.

## Manual workaround (использовался для doставки run #13)

```bash
export TG_BOT_TOKEN="..." TG_BUSINESS_CHAT_ID="986299192" \
       TG_OPS_CHAT_ID="986299192" HC_PING_URL="..."
uv run python -m ga_crawler deliver-run --run-id 13
```

Сработало → msg_id=11 доставлен. Но это не masштабируется.

## Planned fix

**Phase 11 DEPLOY-03**: add `load_dotenv(verbose=True)` at `src/ga_crawler/__main__.py` entrypoint — ДО любого CLI dispatching.

```python
# src/ga_crawler/__main__.py
from dotenv import load_dotenv
load_dotenv(verbose=True)  # idempotent — safe to call multiple times
# ... existing CLI dispatch
```

`load_dotenv()` идемпотентна и не overrides already-set env vars by default. Значит:
- На VPS через bash wrapper: `set -a; source .env` уже выставил vars → `load_dotenv` no-op (existing values preserved)
- На Windows local или manual recovery: vars не выставлены → `load_dotenv` читает `.env` → vars loaded

Должно работать identically для всех code paths.

## Test для prevention

**Phase 11 clean-shell canary test** — pytest test который:
1. Создаёт mock `.env` с fake токенами
2. Запускает `python -m ga_crawler weekly-run --dry-run` через `subprocess.run` с **clean env** (`env={}`)
3. Asserts что Python child видит env vars из mock `.env`
4. Asserts что `delivery_status != skipped_no_credentials`

Ловит recurrence если кто-то снова rip'нет `load_dotenv` ради "оптимизации".
