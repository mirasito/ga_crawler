# v1.1 Findings — Parser bugs discovered in first live-run #13 (2026-05-13)

> Evidence-only doc. Seed for `/gsd-new-milestone` → parser-fix phase planning.
> Discovered when user requested full crawl from Windows local; xlsx report
> came back empty despite 82 viled + 88 goldapple SKU persisted to DB.

## Context

- Run #13 (2026-05-13 12:09 UTC, Windows local, Camoufox 0.4.11)
- Pipeline finished `status=success`, all sanity gates lowered to 1
- Reporter wrote `reports/2026-W20.xlsx` (8KB) with **zero rows in all data sheets**
- Root cause: `match.goldapple_comparable_count=0` → no matched pairs → empty per-sku-deltas and assortment-gaps sheets

## Bug #1 — goldapple `volume_norm` empty for 88/88 SKU

**Evidence (live PDP screenshot, 2026-05-13, brand STEREOTYPE / SKU sago):**

The goldapple PDP renders volume in a **separate structured block**, not embedded
in the product title:

```
[78]  ОБЪЁМ / МЛ
```

Where `78` is the numeric value rendered in a flat-box with the literal label
"ОБЪЁМ / МЛ" beside it. The block sits below the title and above the price.

**Current parser behavior** (from DB sample of run #13 goldapple snapshots):

| sku_id | name | volume_raw | volume_norm | current_price |
|--------|------|------------|-------------|---------------|
| ... | Armaniarmani code | NULL | NULL | 55380 |
| ... | ArmaniARMANI CODE HOMME | NULL | NULL | 40975 |

`volume_raw=NULL` means the parser is **not finding the volume tag at all** —
it isn't even capturing the raw string before normalization.

**Hypothesis (unverified — needs live HTML inspection):** Parser likely uses
microdata or a selector that targeted an older PDP structure. The `75 ОБЪЁМ / МЛ`
block in the 2026-05-13 PDP is rendered as a flexbox of `<div>`s with no
`itemprop="size"` or similar semantic markup → microdata parser comes back
empty. Volume extraction must move to a positional/textual selector.

**Code reference:** `src/ga_crawler/parsers/goldapple_*.py` (whichever owns
volume extraction for Phase 3 microdata path).

## Bug #2 — goldapple `brand` + `name` concatenated/inverted

**Evidence (DB sample, run #13):**

| brand | name (verbatim from DB) |
|-------|-------------------------|
| (empty) | `Armaniarmani code` |
| Armani | `ArmaniARMANI CODE HOMME` |
| (empty) | `Armaniarmani code` |

Observations:
1. The brand "Armani" appears **as a prefix inside the `name` field**, either
   in all-lowercase (`Armaniarmani`) or all-caps (`ArmaniARMANI`)
2. The `brand` column is sometimes empty even though the brand is recoverable
   from the name prefix
3. The actual product name is appended after the brand prefix in mixed casing

**Live PDP evidence:** Title rendered as `STEREOTYPE sago` where:
- `STEREOTYPE` is the brand (uppercase)
- `sago` is the product name (lowercase, prepended-stripped)

The parser appears to concatenate the brand-displayed-as-title with the
product-name-displayed-as-title without separating them, producing
`STEREOTYPESago` or `STEREOTYPEsago` strings.

**Hypothesis:** Parser reads either:
- The full `<h1>` title verbatim into `name` and never extracts the brand from
  the structured product detail block, OR
- Two adjacent text nodes (brand + name) into a single string via `text()`
  without a separator

**Code reference:** Same parser module as Bug #1.

## Bug #3 — viled `volume_raw` = full `name` string

**Evidence (DB sample, run #13 viled):**

| brand | name | volume_raw | volume_norm |
|-------|------|------------|-------------|
| Jo Malone London | Одеколон Myrrh & Tonka Cologne Intense Limited Edition, 100 мл | Одеколон Myrrh & Tonka Cologne Intense Limited Edition, 100 мл | (Decimal('100'), 'ml', 1) |
| Frederic Malle | Парфюмерная вода Contre-Jour | Парфюмерная вода Contre-Jour | NULL |
| Creed | Парфюмерная вода Wild Vetiver | Парфюмерная вода Wild Vetiver | NULL |

`volume_raw` equals the **entire name** string verbatim, not a separately
extracted volume field. The volume normalizer then runs a regex against this
combined string:
- If `"100 мл"` (or similar) appears literally in the name → `volume_norm` is
  populated (e.g. Jo Malone above)
- Otherwise → `volume_norm = NULL` (e.g. Frederic Malle Contre-Jour, Creed Wild
  Vetiver — neither has volume in the displayed name)

This is less broken than goldapple (some volumes DO get extracted), but it
relies on volume being literally embedded in the product title, which is not
guaranteed.

**Better behavior:** Read volume from a dedicated viled PDP field if one exists
(needs spike against live HTML). Otherwise document the limitation explicitly
in the normalizer with an audit log line.

**Code reference:** `src/ga_crawler/parsers/viled_nextdata.py`

## Why v1.0 audit didn't catch this

v1.0 milestone audit verdict was `tech_debt` (paperwork only — SECURITY/VALIDATION
.md missing for phases 2/4/6) per `milestones/v1.0-MILESTONE-AUDIT.md`. The 803
unit-test suite is **all green** but tests parsers against **fixtures captured
at Phase 2/3 build time** (May 5–11). Live PDPs on May 13 already drifted, or
the original fixtures never covered these specific shapes (Armani brand on
goldapple, Contre-Jour on viled). This is a fixture-vs-live drift gap that the
v1 verification workflow did not bridge.

Recommended for v1.1 phase: include **live HTML capture against the production
sites** as part of the parser-fix verification, not just frozen fixtures.
