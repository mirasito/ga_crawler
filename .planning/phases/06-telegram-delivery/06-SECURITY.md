## T-06-01 Verification Detail

**Declared mitigation:** `_esc(value)` wraps `stdlib_html.escape(value, quote=False)` at lines 75-82; applied to every dynamic str field in `build_ops_alert`.

**Grep evidence — `_esc` definition:**

`src/ga_crawler/delivery/message_builder.py` lines 75-82:
```python
def _esc(value: str) -> str:
    """HTML-escape per Telegram HTML parse_mode rules (Pitfall A, caveat #3).
    ...
    """
    return stdlib_html.escape(value, quote=False)
```

**Grep evidence — every interpolation point in `build_ops_alert`:**

| Line | Dynamic field | `_esc()` applied |
|------|---------------|-----------------|
| 115 | `reason_short` | YES — `_esc(reason_short)` |
| 117 | `_format_almaty(started_at_utc)` | YES — `_esc(_format_almaty(...))` |
| 118 | `run_status` | YES — `_esc(run_status)` |
| 119 | `gate_failed_check or ''` | YES — `_esc(gate_failed_check or '')` |
| 132 | `error_short[:truncate_chars]` | YES — `_esc(truncated)` |

Integer/float fields (`viled_count`, `goldapple_count`, `match_count`, `match_rate`, `xlsx_size_mb`, `size_limit_mb`, `run_id`) are not wrapped in `_esc()` because they are numeric types — not str — and cannot carry HTML injection payload. This is correct.

`run_id` at line 137 (manual recovery command) is also an integer; no escaping required.

**Verdict:** Mitigation fully present. T-06-01 CLOSED.

## Accepted Risks

### T-06-02 — Bot token in plaintext `.env`

**Threat:** Bot token stored in `.env` file on disk; file permission misconfiguration could expose it.

**Disposition:** accept

**Rationale:** Token is read exclusively via `os.getenv("TG_BOT_TOKEN")` in `delivery/config.py:97` — never hardcoded in source. `.env` is listed in `.gitignore`; only `.env.example` (with placeholder) is committed. The Python code layer cannot enforce OS filesystem permissions — this is an operator responsibility documented in `README.md:51`:

```
sudo -u ga_crawler chmod 0600 .env   # T-07-08 mitigation — read+write только владельцу
```

OS-level `chmod 0600` limits access to the owning process user (`ga_crawler`). This is the industry-standard treatment for secret token files in single-user VPS deployments. Accept is consistent with Phase 7 `07-SECURITY.md` treatment of the same pattern.

**Residual risk:** An operator who skips the `chmod 0600` step leaves the token world-readable. Mitigated by explicit README documentation. No Python-layer mitigation is possible.

**Accepted by:** Phase 6 threat register (PLAN 06-01); consistent with Phase 7 audit.

### T-06-03 — `business_caption()` `summary_text` not `_esc()`'d

**Threat:** `summary_text` passed verbatim to Telegram `send_message` / `send_document` caption without HTML escaping.

**Disposition:** accept

**Rationale:** `summary_text` is internal pipeline output generated by `reporter/summary_builder.py::build_summary()`. Its data sources are:

1. **Numeric stats** — `viled.fetch_count`, `goldapple.fetch_count`, `match.count`, `match.rate` (integers / floats from `runs.stats`).
2. **Numeric derived values** — `gaps_count`, `promo_count` (integers from DataFrame row counts).
3. **Controlled string constant** — `iso_week` (e.g. `"2026-W19"`, generated from `datetime.isocalendar()`).
4. **Hardcoded D-504 emoji template** — `SUMMARY_TEMPLATE` and `TOP3_LINE` are module-level string constants in `reporter/summary_builder.py`; their shape is git-frozen and regression-tested by the golden-file canary `tests/fixtures/reporter/expected-summary-text.txt`.
5. **Normalised product fields** — `brand_norm`, `name_norm`, `volume_norm` from the matcher's strict-key normalisation pipeline (lowercased, punctuation-stripped). These have been normalised before reaching the summary builder; they are not raw user or scraper input at this layer.

Applying `_esc()` to `summary_text` would break intentional Telegram HTML tags (`<b>`, `<code>`, etc.) that `summary_builder.py` may embed as part of the D-504 template. No external user input is reflected in `summary_text`; the injection surface does not exist.

**D-1002 auto-flip guard (RESEARCH §2 Claim 3):** This threat was pre-classified as `accept` in the threat register specifically to prevent automated reclassification to `mitigate` on the basis of absent `_esc()` coverage. The accept disposition is authoritative.

**Residual risk:** If `summary_builder.py` is modified in a future phase to incorporate external input (e.g., product names from scraped HTML without normalisation), the accept disposition must be re-evaluated and the threat should be promoted to `mitigate` at that time.

**Accepted by:** Phase 6 threat register (PLAN 06-01).
